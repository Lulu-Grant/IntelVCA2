# Node 镜像、启动模式与持久化

## 1. 先理解：启动的不是“显卡功能”，而是节点系统

VCA2 的使用核心不是“驱动装上就能直接用 GPU”，而是：

- 宿主机管理卡
- 每个节点单独启动自己的系统
- 实际操作对象是节点里的操作系统环境

因此镜像和启动方式就是关键。

## 2. 两种主流启动模式

现有资料反复提到 VCA2 常见的两条路线：

- RAMDisk 模式
- BlockIO 模式

## 3. RAMDisk 模式

### 3.1 原理

大致过程是：

1. 宿主把某个节点和镜像绑定
2. 节点启动时把镜像注入内存
3. 节点把部分内存当作系统盘来运行

### 3.2 优点

- 官方环境往往更完整
- 更适合快速验证“节点能不能启动”
- 某些场景下上手更快

### 3.3 缺点

- 本质上跑在内存里
- 一旦重启或断电，数据会丢
- 后续环境维护依赖自动化脚本和外部存储

### 3.4 适合什么场景

- 第一次验证卡是否点亮
- 只想测试某个节点能否成功启动
- 临时实验

## 4. BlockIO 模式

### 4.1 原理

BlockIO 模式的思路更像虚拟机磁盘：

- 宿主机保存节点镜像文件
- 把镜像文件映射给节点
- 节点把它当成真实系统盘使用

### 4.2 优点

- 节点重启后数据仍然保留
- 适合长期维护和持续折腾
- 更接近“真正能用的系统环境”

### 4.3 缺点

- 前期镜像准备更重要
- 更依赖宿主机存储规划

### 4.4 适合什么场景

- 长期保留节点环境
- 部署媒体服务
- 持续修改系统配置
- 做多节点实验

## 5. 两种模式怎么选

一个简单判断表：

| 目标 | 推荐模式 |
| --- | --- |
| 先确认节点能不能跑起来 | RAMDisk |
| 想长期保留环境和数据 | BlockIO |
| 想部署服务并重复使用 | BlockIO |
| 只做临时点火测试 | RAMDisk |

如果没有特别明确的限制条件，公开资料更推荐：

> **优先使用 BlockIO 作为主玩法。**

## 6. 镜像准备时该关注什么

无论哪种模式，节点镜像都应关注：

- 来源是否匹配当前使用的驱动版本
- 是否包含节点需要的基础系统组件
- 镜像对应的是 Linux 还是 Windows
- 是否适合长期写入、更新和备份

由于当前资料没有给出统一、可验证的镜像命名规则和命令参数，因此这里不写死具体命令，而是建议优先记录：

- 镜像文件名
- 来源
- 对应驱动版本
- 用途（测试 / 持久化 / Windows / Linux）

## 7. M.2 固态与镜像的关系

资料里一个比较明确的共识是：

- 卡上的 M.2 固态不是必须
- 节点镜像可以放在宿主的其他存储里
- M.2 更像可选增强项

因此可以这样理解：

- 没有 M.2：照样能玩
- 有 M.2：迁移、随卡带盘更方便

## 8. 节点启动后的完成标志

无论用哪种模式，启动成功后的关键不是“命令跑完了”，而是：

- 节点在管理工具里状态正常
- 节点拿到可登录的系统环境
- 后续可以继续完成网络与 SSH 访问配置

如果启动后仍旧不能进入后续使用阶段，优先检查：

- 镜像是否匹配
- 节点内存是否正常
- 启动模式是否选错
- 驱动与镜像版本是否对应

## 9. Windows 镜像怎么理解

资料提到官方存在 Windows 方案，但并不将其作为主流路径。

更合理的态度是：

- 可以研究
- 不适合作为首选路线
- 真正有价值的场景仍然主要在 Linux 与视频处理链路

## 10. 持久化建议

如果目标是长期保留某个节点环境，建议至少做到：

- 优先用 BlockIO
- 定期备份镜像文件
- 把节点用途固定下来，不要频繁改成完全不同的工作负载
- 每个节点分工清楚，例如：
  - Node 1：测试
  - Node 2：媒体服务
  - Node 3：实验 / 备份

这样后面排错和迁移会轻松很多。

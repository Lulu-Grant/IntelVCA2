# 网络、SSH 与服务部署

## 1. 节点为什么没有网口却能联网

VCA2 的节点不是通过实体网口接入网络，而是：

- 节点与宿主之间通过 **PCIe 总线** 建立内部通信
- 宿主机上出现或配置出面向节点的内部网络
- 宿主机再作为网关，对节点做 NAT 或转发

对使用者来说，更容易理解成：

> 节点和宿主之间有一条“非常快的内部虚拟网络”。

## 2. 这条内部网络有多快

现有资料给出过 iPerf3 测试结果：

- 正向约 `18.1 ~ 18.8 Gbps`
- 反向接近 `32 Gbps`

这说明：

- 宿主与节点之间的数据交换带宽很高
- 本地服务访问、镜像读写、媒体流转都能受益

## 3. 基本网络思路

节点联网的完整逻辑通常是：

1. 节点通过内部网络与宿主机通信
2. 宿主机作为节点的默认网关
3. 宿主机把节点转发到外部网络

如果只做到这一步，通常意味着：

- 节点可以主动访问外网
- 但外界不一定能主动访问节点服务

## 4. 如果要从外部访问节点服务

你通常还需要在宿主机上处理：

- 端口映射
- DNAT / 转发
- 防火墙规则

否则会出现一种常见情况：

- 节点里服务已经跑起来了
- 节点自己也能联网
- 但手机、PC 或外部客户端连不过去

## 5. SSH 登录节点

资料明确指出，节点本质上是无头服务器：

- 没有视频输出
- 不靠显示器管理
- 主要通过 SSH 使用

典型流程是：

1. 在宿主机上查到节点 IP
2. 使用 SSH 连接节点
3. 用镜像默认账号登录

示意形式通常是：

```bash
ssh <user>@<node-ip>
```

## 6. 默认账号密码怎么处理

这里必须谨慎，因为现有资料对默认账号记载并不完全一致。

能相对确定的是：

- 默认密码常被记为 `vista1` / `Vista1`
- 默认账号在不同资料里记录并不完全统一

因此更稳妥的做法是：

- 先查你手头镜像或驱动包附带说明
- 第一次登录成功后立刻改密码
- 不要把默认口令长期保留

## 7. 登录后应该先检查什么

建议进入节点后优先确认：

- `cat /proc/cpuinfo`：CPU 型号是否正确
- `free -h`：内存容量是否符合预期
- `df -h`：系统盘是否正常显示
- 网络是否能连到宿主或外网

如果这些基础项都对，说明节点环境已经进入可用状态。

## 8. Jellyfin / Docker 路线怎么理解

资料里最明确的实战用途就是：

- 在节点内安装 Docker
- 运行 Jellyfin
- 利用节点的 CPU 与 P580 做媒体转码

重要的是理解它的原理，而不是机械抄命令：

- 节点是老系统
- 容器环境也要服从老系统约束
- 图形 / 视频设备需要能被容器访问

因此你真正要确认的是：

- 容器运行环境是否兼容
- 节点里的图形设备是否已正确暴露
- Jellyfin 是否启用了正确的硬件加速策略

## 9. 10-bit HEVC 是一个关键限制

资料里最重要的一条实战结论是：

- **P580 不支持 10-bit 解码**

因此如果你在 Jellyfin 里直接勾选 10-bit 硬件解码，常见结果就是：

- 视频无法正常转码播放

更合理的策略是：

- CPU 负责 10-bit 软解
- GPU 负责后续硬件编码输出

也就是常说的：

> CPU 软解 + GPU 硬编码

## 10. 这种混合模式为什么有效

资料里给出的解释是：

- 节点 CPU 有一定软解能力
- 处理器带有 128MB eDRAM
- P580 可接手后续编码工作

实际表现为：

- CPU 占用很高
- GPU 频率也明显拉升
- 但视频拖动与恢复画面速度依然可接受

## 11. 实际适合部署哪些服务

更适合：

- Jellyfin
- 轻量媒体服务
- 视频批处理 / 代理转码
- 多节点隔离实验环境

不太适合：

- 现代高并发公网主力服务
- 需要最新视频格式完整支持的方案
- 面向公网直接裸露的老系统服务

## 12. 服务部署建议

如果你要长期用，建议这样分工：

- 宿主机：只负责卡、网络、镜像和转发
- 节点：只负责具体服务
- 外网暴露：尽量由更现代、更安全的一层代理处理

这会比“老宿主直接暴露公网”安全得多。

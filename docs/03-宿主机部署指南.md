# 宿主机部署指南

## 1. 宿主机的角色是什么

在 VCA2 架构里，宿主机不是“单纯把卡插上去的电脑”，它要承担这些职责：

- 提供兼容的物理平台
- 为 VCA2 提供电力、散热、PCIe 资源
- 运行宿主驱动和节点管理工具
- 存放节点镜像
- 作为节点网络的网关
- 必要时对外做端口映射

也就是说：

> 宿主机本身就是 VCA2 系统的一部分。

## 2. 推荐部署思路

基于现有资料，较稳妥的总体路线是：

1. 在物理机上安装老版本宿主系统
2. 插卡、上电、确认硬件识别
3. 安装宿主驱动与节点管理工具
4. 确认三个节点状态正常
5. 准备节点镜像
6. 选择 RAMDisk 或 BlockIO 启动方式
7. 配置宿主与节点之间的内部网络

## 3. 宿主机系统建议

资料里最常见、成功率最高的路线是：

- **CentOS 7.6 物理机**

原因不是“最好用”，而是：

- 更接近原始驱动生态
- 与流传资料更匹配
- 更容易把节点真正拉起来

不建议一上来就尝试：

- 现代发行版硬怼
- PVE 直通
- 宿主虚拟化再套 VCA2

## 4. 硬件安装流程

建议按照下面顺序做：

1. 先确认宿主机平台与 BIOS 资源配置没有明显限制
2. 按需给节点插入 DDR4 SO-DIMM
3. 将 VCA2 插入合适的 PCIe 插槽
4. 接好 8Pin + 6Pin 供电
5. 确认风道或外加风扇已经到位
6. 再开机进入宿主系统

补充说明：

- 每个节点相互独立，不用的节点可不插内存
- 内存规划会直接影响你后续给节点跑什么任务

## 5. 宿主机第一步验证

进入宿主系统后，先做的是**确认硬件层面被识别**，而不是急着装服务。

现有资料最明确提到的验证命令是：

```bash
lspci
```

你的目标不是看到某个固定字符串，而是确认：

- VCA2 相关设备确实出现在 PCIe 设备列表中
- 宿主没有在更早阶段就识别失败

如果这一步就不对，优先怀疑：

- 主板兼容性
- 插槽位置
- 供电
- 风道以外的硬件稳定性
- BIOS / 资源分配问题

## 6. 驱动安装阶段该怎么理解

现有资料都强调了一点：

- 在环境对齐之前，驱动很难
- 在环境对齐之后，驱动反而不一定复杂

因此正确姿势是：

- 先保证平台和系统版本正确
- 再按驱动包内文档执行安装

这里不在本文里硬写具体命令，原因是：

- 不同流传包版本可能不同
- 安装脚本、服务名、目录结构可能不同

因此更可靠的做法是：

1. 对照驱动包说明
2. 记录自己当前使用的驱动版本
3. 把宿主机安装过程做成自己的部署笔记

## 7. 安装完成后要验证什么

驱动装完不等于成功。真正关键的是：

- 节点管理工具是否工作
- 三个节点是否都能正常显示状态

只有当你能在宿主机侧看到：

- 节点存在
- 节点状态可读
- 节点可进入后续启动流程

才说明整张卡真正“活过来了”。

## 8. 宿主侧存储建议

即使卡上有 M.2，当前资料也明确指出：

- 节点镜像不强制必须放在卡载 M.2 上
- 完全可以放在宿主机自己的硬盘里

因此更实际的宿主存储规划是：

- 系统盘：宿主机自身系统
- 数据盘：节点镜像、备份、日志
- 可选盘：卡载 M.2，用于随卡迁移或专用镜像存放

## 9. 为什么宿主机最好只做“底座”

因为宿主机通常需要跑：

- 老系统
- 老驱动
- 老工具链

这类环境天然不适合直接对公网暴露，也不适合承载太多现代业务。

比较稳妥的思路是：

- 宿主机负责把 VCA2 伺候好
- 业务尽量在节点里，或在宿主里的现代虚拟机里完成

## 10. 这一步的完成标准

你可以把宿主机部署完成理解为以下条件都成立：

- 卡已上电并稳定运行
- 宿主系统是兼容版本
- 宿主能识别到 VCA2 相关硬件
- 驱动与节点管理工具已安装
- 三个节点状态可见
- 宿主已有用于后续镜像和网络配置的基础环境

达成这一步后，才值得继续折腾镜像、启动和服务部署。
